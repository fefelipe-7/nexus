# nexus context

Esta pasta contém o contexto oficial do projeto nexus.

O objetivo dela é permitir que IAs, agentes de desenvolvimento e contribuidores humanos entendam o projeto de forma consistente, sem depender de conversas anteriores.

O nexus é um app pessoal local-first e offline-first para organizar, armazenar, relacionar e analisar dados importantes da vida do usuário.

O nome oficial do app é sempre nexus, em letras minúsculas.

## função desta pasta

A pasta .context funciona como a memória oficial do projeto.

## regra mais importante do projeto

Submódulos são fontes reais de entrada de dados.

Visualizações, widgets, indicadores e insights são resultados calculados.

Regra central:

submódulo = input
visualização = output

Se algo recebe dados diretamente, pode ser submódulo.

Se algo apenas calcula, resume, interpreta ou cruza dados existentes, deve ser visualização, widget, indicador ou insight.

## módulos principais

O nexus possui 5 módulos principais:

1. saúde
2. mente
3. ação
4. finanças
5. vida

Cada módulo agrupa submódulos que recebem dados reais.

## visão resumida

O nexus não é apenas um tracker.

Ele é um sistema pessoal de evolução.

O app deve ajudar o usuário a registrar dados da vida, conectar informações entre áreas diferentes e entender sua evolução real ao longo do tempo.

## princípios centrais

O nexus deve ser:

- local-first;
- offline-first;
- privado por padrão;
- modular;
- relacional;
- otimizado para mobile;
- preparado para grande volume de dados locais;
- orientado a evolução pessoal;
- assistido por IA, mas não dependente dela.

## decisões já assumidas

O projeto já assume que:

- o app se chama nexus;
- o nome deve ser escrito em letras minúsculas;
- o app terá 5 módulos principais;
- os módulos são saúde, mente, ação, finanças e vida;
- submódulos são fontes reais de dados;
- visualizações são resultados calculados;
- o app será local-first;
- o app será offline-first;
- o banco local será SQLite;
- o app usará Kotlin Multiplatform;
- o backend será complementar;
- a sincronização será própria e local-first;
- IA será importante, mas não deve ser dependência obrigatória.

App:

- Compose Multiplatform

- Kotlin Multiplatform

Backend planejado:

Mudanças pequenas podem ser feitas diretamente nos arquivos relevantes.

Mudanças grandes devem gerar ou atualizar um ADR.

Exemplos de mudanças que exigem ADR:

- mudar stack principal;
- mudar banco local;
- mudar estratégia local-first;
- mudar separação entre submódulos e visualizações;
- mudar modelo de sync;
- mudar arquitetura principal;
- adicionar dependência obrigatória de nuvem;
- alterar princípios de privacidade.
